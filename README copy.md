# repo_standardizer Skill（企业级架构增强版 v3）

repo_standardizer 是一个企业级 ClaudeCode
Skill，用于自动将任意代码仓库重构为符合：

-   Clean Architecture（整洁架构）
-   DDD（领域驱动设计）
-   分层架构（Layered Architecture）
-   模块化架构（Modular Architecture）
-   AI Agent 原生架构（Agent-native Architecture）
-   测试隔离规范（Test Isolation Standard）

的标准化生产级代码库。

它不仅重构代码结构，更会强制执行架构规则。

------------------------------------------------------------------------

# 适用场景

-   混乱仓库重构
-   单文件项目拆分

-   团队工程标准统一

# 核心能力

## 1️⃣ 自动架构标准化

将任意仓库重构为：

-   分层架构
-   模块化结构
-   清晰依赖关系
-   可维护工程结构

自动消除：

-   单体文件（monolith）
-   结构混乱
-   循环依赖
-   职责混乱
-   根目录污染

------------------------------------------------------------------------

## 2️⃣ 强制拆解单体脚本（Monolith Decomposer）

自动检测：

-   300行文件

-   多职责文件

-   混合 models / logic / DB / IO

-   单文件项目

并强制拆解为：

    models/
    services/
    repositories/
    handlers/
    utils/
    configs/
    agents/

原始入口文件将只保留：

-   初始化
-   依赖注入
-   启动逻辑

禁止保留业务逻辑。

------------------------------------------------------------------------

## 3️⃣ 测试隔离规范（Test Isolation Rule）

测试文件：

-   不允许出现在根目录
-   不允许与源码混放

自动执行：

-   创建 `tests/`
-   移动所有测试文件
-   修复 import
-   保持逻辑分组

支持识别：

    test_*.py
    *_test.py
    *.spec.ts
    *.test.ts
    *.spec.js
    *.test.js
    *_test.go
    *Test.java
    *_spec.rs

------------------------------------------------------------------------

# 强制执行架构规则

## 🚫 禁止

禁止在以下位置编写业务逻辑：

-   main.py
-   handlers/
-   controllers/
-   routes/
-   根目录脚本

------------------------------------------------------------------------

## ✅ 业务逻辑必须位于

    services/

------------------------------------------------------------------------

## ✅ 数据访问必须位于

    repositories/

------------------------------------------------------------------------

## ✅ 数据结构必须位于

    models/

------------------------------------------------------------------------

## ✅ Agent逻辑必须位于

    agents/

------------------------------------------------------------------------

## ✅ 测试必须位于

    tests/

------------------------------------------------------------------------

# 严格依赖方向

依赖关系必须为：

    handlers
       ↓
    services
       ↓
    repositories
       ↓
    models

禁止：

-   反向依赖
-   循环依赖
-   跨层跳跃依赖
